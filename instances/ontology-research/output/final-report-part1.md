# 字段级 Ontology Graph 研究报告（上）

# 执行摘要与核心发现

> **项目名称**：LLM + Code Graph → 字段级 Ontology Graph  
> **目标**：自底向上构建全公司字段级数据语义图谱，替代传统 top-down 数据分类  
> **状态**：研究完成，含 PoC 验证代码，建议启动 Phase 1 (PoC)

## 1. 执行摘要

> **TL;DR**：用 LLM 分析源代码语义 + Code Graph 调用链，自动构建覆盖 10 万微服务、2000 万字段的 ontology 图谱。首年投入 ~$418K（85% 是人力），3 年 ROI 313%。PoC 可在 2 个月内用 5 个服务验证核心假设。

### 一句话结论

**我们可以用 LLM 从代码中自动发现"这个字段是什么、从哪来、到哪去、是否敏感"，精度 92-97%，成本仅为传统方案的 1/4。**

### 关键数字

| 指标 | 数值 |
|------|------|
| 覆盖范围 | 10 万服务、200 万 API、2000 万字段 |
| 预估图谱规模 | ~3700 万节点、~1.27 亿条边 |
| LLM 全量扫描成本 | **~$680**（分层策略：90% 小模型 + 10% 大模型） |
| 首年总投入 | ~$418K（含开发人力 $360K） |
| 年度运维成本 | ~$176K |
| 3 年 ROI | **313%**（vs 传统方案 $3.18M） |
| 预期精度 | 92-97%（含人工审核兜底） |
| PoC 周期 | 2 个月，5 个服务 |

### 给决策者的建议

**建议立项**。理由：
1. LLM 成本已降至可忽略水平（全量扫描 $680），人力才是大头
2. 代码是"最真实的文档"——从代码提取比人工标注更准确、更及时
3. 合规（PIPL/GDPR）和安全是刚需，且是最容易量化 ROI 的场景
4. PoC 投入极小（2 人 × 2 月），失败代价低

## 2. 项目背景与问题定义

> **TL;DR**：传统数据治理依赖人工上报，覆盖率 ~30%。我们探索从代码自动发现数据语义的新路径。

### 2.1 为什么需要数据治理？

字节跳动的微服务架构已发展到 **10 万+** 服务规模。数据（用户手机号、余额、身份证号等）在这些服务之间流转，但没有人能回答：

- "用户手机号经过了哪些系统？" → 合规审计需要
- "哪些外部 API 暴露了数据库中的 PII？" → 安全需要
- "修改这个字段会影响哪些下游？" → 开发需要

### 2.2 传统方案为什么不够？

| 问题 | 传统方案 | 实际效果 |
|------|---------|---------|
| 数据分类 | 人工标注 + 团队上报 | 覆盖率 ~30%，且快速过时 |
| 数据血缘 | Hive/Spark ETL lineage | 只覆盖数据仓库，不覆盖微服务代码 |
| 合规审计 | 人工追查代码 | 响应时间数周，依赖开发者记忆 |
| 商业产品 | Collibra ($170K+/年) | Top-down 模式，依赖人工维护术语表 |

**核心矛盾**：传统方案是 **top-down**（先定义标准 → 再映射到数据），在 10 万服务的规模下不可行。

### 2.3 我们的假设

**代码是最可靠的数据字典。** 如果能用 LLM 理解代码语义，就能自底向上地自动构建数据分类——无需人工上报。

## 3. 核心技术方案概述

> **TL;DR**：5 层 pipeline——AST 解析 → 预处理 → LLM 语义提取 → 调用链传播 → 图存储查询。批量 + 增量双模式。

### 3.1 五层 Pipeline 架构

```
Layer 1: 数据采集层
  Git 仓库 → Thrift IDL + Go Handler + GORM Model
  Code Graph API → 函数级调用链
  已有 DB 标签 → 存量分类标注

Layer 2: 预处理层
  Go AST 解析 → struct 定义、函数签名、import 关系
  IDL 解析 → 接口定义、字段类型、required/optional

Layer 3: LLM 语义提取层（核心）
  分层策略：
    ├─ 90% 字段 → 小模型 (GPT-4o-mini, $0.15/M tokens)
    └─ 10% 复杂字段 → 大模型 (GPT-4o, $2.50/M tokens)
  输出：字段语义描述（中文）、敏感性分类、置信度

Layer 4: 语义传播层
  正向传播：API 字段 → 调用链 → DB 列
  反向传播：DB 标签 → 调用链 → API 字段
  置信度：Noisy-OR 多路径合并，4 种映射类型衰减

Layer 5: 图存储与查询层
  PoC: Neo4j Community → 试点: NebulaGraph → 全量: ByteGraph
  Cypher 查询 API + REST Gateway
```

### 3.2 关键设计决策

| 决策 | 选择 | 理由 |
|------|------|------|
| 图模型 | Property Graph（非 RDF） | 需要边属性（置信度、传播类型），团队工程导向 |
| LLM 策略 | 分层（小模型+大模型） | 成本降 65%，精度仅损失 3-5% |
| 传播算法 | BFS + 置信度衰减 | 简单可靠，4 种映射类型覆盖 95% 场景 |
| Code Graph 降级 | Go AST 自建 | 60-70% 覆盖率，保证不强依赖单一平台 |

## 4. 关键创新点

> **TL;DR**：业界唯一从源代码出发、用 LLM 自动构建字段级数据语义的方案。

### 4.1 与现有方案的定位差异

```
                    覆盖层
                    ↓
  代码层            数据仓库层          网络层
  ┌──────────┐    ┌──────────────┐   ┌──────────┐
  │ 我们 ⭐   │    │ OpenLineage  │   │ 传统 DLP  │
  │ 唯一覆盖  │    │ DataHub/Atlas│   │          │
  │ 代码语义  │    │ Collibra     │   │          │
  └──────────┘    └──────────────┘   └──────────┘
```

### 4.2 五个差异化优势

| # | 优势 | 说明 |
|---|------|------|
| 1 | **代码级分析** | 唯一从源代码出发的方案，其他都从"数据已在系统中"开始 |
| 2 | **LLM 语义理解** | 不是正则匹配，能理解 "uid 在直播场景中代表观众 ID" |
| 3 | **全链路追踪** | API 入口 → 函数调用链 → DB 存储，跨服务追踪 |
| 4 | **Bottom-up 自动发现** | 不依赖人工定义术语表，直接从代码发现 |
| 5 | **Git push 实时更新** | 代码提交即更新，比定期爬取快几个数量级 |

### 4.3 诚实的劣势

| 劣势 | 严重性 | 缓解 |
|------|--------|------|
| 全新方案，无生产验证 | 🔴 高 | 分阶段 PoC → 试点 → 全量 |
| LLM 精度非 100% | 🟠 中 | 分层策略 + 人工审核，综合 92-97% |
| 不覆盖数据仓库层 | 🟠 中 | 与 OpenLineage/DataHub 互补 |
| 依赖 Code Graph | 🟡 中 | AST 降级方案覆盖 60-70% |

## 5. 落地价值分析

> **TL;DR**：7 个落地场景，最高价值的 3 个（数据分类自动化、合规分析、安全发现）直接对应合规和安全刚需。

### 5.1 七大场景概要

| 优先级 | 场景 | 核心价值 | 量化指标 |
|--------|------|---------|---------|
| 🔴 P0 | **数据分类自动化** | 自动发现全公司 PII/Financial 字段 | 覆盖率 30% → 90% |
| 🔴 P0 | **合规隐私分析** | 回答"手机号经过了哪些系统" | 审计响应 数周 → 小时 |
| 🔴 P0 | **安全风险发现** | 发现 API 暴露的敏感数据 | 单次发现可避免 ¥100 万+ 罚款 |
| 🟠 P0.5 | **变更影响评估** | CI/CD 自动拦截高风险变更 | 减少 50%+ 数据事故 |
| 🟡 P1 | **API Schema 语义** | 统一字段语义描述 | 减少 30-50% 跨团队沟通 |
| 🟡 P1 | **架构可视化** | 字段级数据流全景图 | 评审效率 3-5x |
| 🟡 P1 | **字段标准化** | 发现同义不同名字段 | API 命名统一 |

### 5.2 最大卖点

**场景 1+2+3 直接对应合规和安全——这是企业最愿意为之买单的痛点。**

一个典型的数据泄露事件成本：
- 直接罚款（PIPL）：¥100 万 - 5000 万
- 声誉损失：难以量化但远超罚款
- 修复成本：全员排查可能花费数月

Ontology Graph 能在数据泄露**之前**发现暴露面。

## 6. 成本与 ROI

> **TL;DR**：首年 ~$418K（85% 人力），3 年 $770K。对比传统方案 $3.18M，节省 75%+，ROI 313%。

### 6.1 成本构成

| 成本项 | 首年 | 年度运维 | 3 年总计 |
|--------|------|---------|---------|
| **人力**（开发+运维+审核） | $360K | $150K | $660K |
| **LLM API** | $11K | $10.4K | $31.8K |
| **基础设施** | $36K | $12K | $60K |
| **培训+差旅** | $11K | $3K | $17K |
| **合计** | **$418K** | **$176K** | **$770K** |

### 6.2 LLM 成本不是瓶颈

- 全量扫描（10 万服务）：**~$680**（一次性）
- 年增量更新：**~$10.4K**
- LLM 仅占总成本 **< 5%**
- **人力才是大头（85%）**——这才是应该优化的方向

### 6.3 ROI 对比

| 方案 | 3 年成本 | 覆盖率 | 实时性 |
|------|---------|--------|--------|
| 传统人工治理 | **$3.18M** | ~30% | 季度更新 |
| 商业产品 (Collibra) | **$510K+** | ~50% | 人工触发 |
| **我们的方案** | **$770K** | ~90% | Git push 实时 |

**ROI = (传统成本 - 我们的成本) / 我们的成本 = ($3.18M - $770K) / $770K = 313%**
