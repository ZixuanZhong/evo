## Phase 1: PoC 验证（第 1-2 月）

### 目标

用 5 个真实微服务验证核心假设：LLM 能否从 Go-Kitex 代码中准确提取字段语义？

### 关键里程碑

| 周 | 里程碑 | 产出 |
|----|--------|------|
| W1 | 环境搭建 | Neo4j Docker 运行 + Python pipeline 骨架 |
| W2 | AST 预处理 | Go AST 解析器，提取 5 个服务的 struct/函数/IDL |
| W3 | LLM 提取 | Prompt 设计完成，5 个服务的字段语义提取 |
| W4 | 语义传播 | 正向/反向传播算法 PoC，3 条完整调用链验证 |
| W5-6 | 图存储+可视化 | Neo4j 写入 + 基础可视化 demo |
| W7-8 | 精度评估+报告 | 人工标注 ground truth 对比，精度报告 |

### 资源需求

| 资源 | 需求 | 备注 |
|------|------|------|
| 后端工程师 | 2 人 | Pipeline 开发 |
| 算法工程师 | 1 人（兼职） | LLM prompt 设计 |
| 基础设施 | 1 台开发机 (32GB RAM) | Neo4j + Python |
| LLM API 预算 | ~$50 | 5 个服务很少 |
| 代码访问权限 | 5 个服务的 Git 仓库读权限 | 选直播业务线的典型服务 |

### 风险与缓解

| 风险 | 概率 | 影响 | 缓解 |
|------|------|------|------|
| LLM 提取精度 < 80% | 中 | 高 | 优化 prompt + 增加 few-shot + 试多个模型 |
| Go AST 解析复杂度超预期 | 低 | 中 | 用 go/ast 标准库，避免过度工程化 |
| 跨服务调用链数据不全 | 中 | 中 | PoC 阶段用 mock 数据补充 |
| 团队对方案信心不足 | 低 | 高 | 尽早产出可视化 demo，用直观结果说服 |

### Go/No-Go 决策点 1（M2 结束）

| 评估标准 | 通过阈值 |
|----------|---------|
| 字段语义提取精度（vs 人工标注） | ≥ 80% |
| 敏感性分类精度（PII/Financial） | ≥ 75% |
| 端到端 pipeline 可运行 | 5 个服务完整跑通 |
| 可视化 demo 可展示 | 有至少 1 个直观的 graph 可视化 |
| 团队和 sponsor 认可 | Go 决定 |

**Go** → 进入 Phase 2（试点）  
**No-Go** → 分析失败原因，调整方案或终止项目

## Phase 2: 试点扩展（第 3-6 月）

### 目标

扩展到 100 个服务，集成 Code Graph 真实数据，建立人工审核闭环，验证规模化可行性。

### 关键里程碑

| 周 | 里程碑 | 产出 |
|----|--------|------|
| W9-10 | Code Graph 对接 | REST API 集成，5 个服务调用链数据 |
| W11-12 | 扩展到 50 个服务 | 批量扫描 pipeline，分层模型策略上线 |
| W13-14 | 人工审核流程 | 低置信度字段审核队列 + 审核 UI |
| W15-16 | 扩展到 100 个服务 | Code Graph 增量推送接入 |
| W17-18 | DB 标签回溯 | 反向传播上线，PII 暴露面报告 |
| W19-20 | 增量更新 | Git webhook 触发增量分析 |
| W21-24 | 精度验证+报告 | 100 服务精度评估，试点总结报告 |

### 资源需求

| 资源 | 需求 | 备注 |
|------|------|------|
| 后端工程师 | 2 人 | 全职 |
| 算法工程师 | 1 人 | 全职 |
| 数据工程师 | 1 人 | 图存储 + Code Graph 集成 |
| 人工审核员 | 1 人 | 低置信度字段审核 |
| PM/架构师 | 0.5 人 | 跨团队协调 |
| Neo4j Enterprise 或 NebulaGraph | 3 节点集群 | 16GB RAM × 3 |
| LLM API 预算 | ~$200 | 100 个服务 |
| Code Graph API 权限 | 试点服务范围 | 与 Code Graph 团队协商 |

### 风险与缓解

| 风险 | 概率 | 影响 | 缓解 |
|------|------|------|------|
| Code Graph API 不可用或延迟 | 中 | 高 | 降级到 AST 自建（覆盖率降低） |
| 100 服务扫描成本超预算 | 低 | 中 | 分层策略 + 预算上限控制 |
| 精度随规模下降 | 中 | 高 | 持续优化 prompt，增加术语表，人工反馈循环 |
| 人工审核成为瓶颈 | 中 | 中 | 优先审核高敏感字段，其余批量接受 |
| Code Graph 团队排期冲突 | 中 | 中 | 提前沟通，提供备选方案（批量 dump） |

### Go/No-Go 决策点 2（M6 结束）

| 评估标准 | 通过阈值 |
|----------|---------|
| 字段语义提取精度 | ≥ 85%（100 服务范围） |
| PII 暴露面发现率 | 发现 ≥ 10 个真实高风险暴露 |
| 增量更新延迟 | < 10 分钟（Git push 到图更新） |
| Code Graph 集成 | 稳定运行 2 周+ |
| 合规/安全团队认可 | 至少 1 个下游团队愿意使用数据 |
| 成本在预算范围内 | 累计 LLM < $500，计算 < $2K |

## Phase 3: 全量部署（第 7-12 月）

### 目标

全公司 10 万服务全量扫描，迁移到 ByteGraph，与下游系统集成，进入稳态运维。

### 关键里程碑

| 月 | 里程碑 | 产出 |
|----|--------|------|
| M7 | ByteGraph 环境准备 | Schema 创建 + 适配器开发 |
| M8 | 全量批处理 | 10 万服务初始扫描（~30 小时） |
| M9 | ByteGraph 迁移 | Neo4j → ByteGraph 数据迁移 + 双写验证 |
| M10 | 下游系统集成 | REST API Gateway + 合规审计 dashboard |
| M11 | CI/CD 集成 | PR 自动风险评论 + 变更影响评估 |
| M12 | 稳态运维交付 | 运维文档 + 监控告警 + SLA 定义 |

### 资源需求

| 资源 | 需求 | 备注 |
|------|------|------|
| 后端工程师 | 2 人 | 全职 |
| 数据工程师 | 1 人 | ByteGraph 迁移 |
| 人工审核员 | 2 人 | 全量扫描审核量大 |
| 运维 | 0.5 人 | 开始建立运维能力 |
| ByteGraph 集群 | 内部资源申请 | ~100GB 存储 |
| LLM API 预算 | ~$680（全量）+ ~$5K（半年增量） | |

### Go/No-Go 决策点 3（M12 结束）

| 评估标准 | 通过阈值 |
|----------|---------|
| 服务覆盖率 | ≥ 80% 的服务成功扫描 |
| 综合精度 | ≥ 88%（含人工审核后） |
| 系统可用性 | ≥ 99.5%（查询 API） |
| 下游使用者 | ≥ 3 个团队在用 ontology 数据 |
| 增量更新成功率 | ≥ 95% |

## 小结

| 维度 | Phase 1 (PoC) | Phase 2 (试点) | Phase 3 (全量) |
|------|--------------|---------------|---------------|
| **范围** | 5 个服务 | 100 个服务 | 10 万服务 |
| **时长** | 2 个月 | 4 个月 | 6 个月 |
| **核心验证** | LLM 提取精度 | 规模化可行性 | 生产级稳定性 |
| **图存储** | Neo4j Community | Neo4j/NebulaGraph | ByteGraph |
| **Code Graph** | Mock 数据 | API 试点接入 | 全量 + 增量 |
| **关键风险** | 精度不足 | 集成延迟 | 运维复杂度 |
| **Go/No-Go** | 精度≥80% | 精度≥85% + 真实发现 | 覆盖≥80% + 3个下游用户 |

**核心原则**：每个阶段都有明确的 Go/No-Go 决策点。**如果 PoC 精度 < 80%，不进入试点**。用最小代价验证最大风险。
